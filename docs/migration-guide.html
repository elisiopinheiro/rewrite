▶ TL;DR

clusters-4wm is being replaced by a clean rewrite, clusters-metadata-api (new /v3 API + SDK). Because consumers migrate over months, both apps must run at the same time against the same PostgreSQL database. We make that work with three Git artifacts and one clever alignment: a one-time compatibility migration (73aee50345b2) reshapes the shared schema and patches the legacy app to match; the rewrite ships a single squashed baseline migration whose id equals that production head, so deploying the rewrite runs no migration on the live DB. The same Helm chart deploys both the internal and partner apps — pointed at the existing RDS. After cutover the rewrite owns the schema and legacy is frozen.

Part I
Overview
Why we're doing this, the three Git artifacts, and how two services share one database.

01Why the rewrite

The legacy clusters-4wm service accumulated patterns that were costly to evolve: model classes that were simultaneously the database table, the validator and the API schema; data access mixed into routers; two parallel API versions (/v1 + /v2); and no enforced quality tooling. The domain itself is small — CRUD over cluster metadata — so a clean rewrite was faster and safer than untangling the old codebase in place. The rewrite also takes a new canonical identity:

Area Legacy New
Service / chart clusters-4wm clusters-metadata-api
API surface /v1, /v2 /v3
Generated SDK clusters-4wm-client clusters-metadata-sdk
Chart version 1.5.0 3.0.0

The hard constraint: there is no big-bang switch. Consumers move in waves, so legacy and new must run simultaneously against one shared database and stay mutually compatible the whole time. Everything that follows is a consequence of that one rule.

03The service & its API surface

What the application actually is, before we get to how it migrates.

clusters-metadata-api is a CRUD service over cluster metadata — clusters and the things attached to them (features, locks, operations, releases, namespaces, node pools, webhooks…). One codebase ships two FastAPI apps, both under the /v3 prefix:

Internal API — app.main:app

/v3/clusters CRUD · status
/v3/features list · create · delete
/v3/clusters/{name}/lock acquire / release
/v3/clusters/{name}/operations operation log
/v3/releases list · get · create · delete

Partner API — app.partner:app

/v3/partners/scp/clusters SCP · CF view
/v3/partners/solar/clusters SOLAR view
read-only role-filtered
{ count, items } uniform envelope

The internal API is the full read/write management surface used by 4Wheels tooling. The partner API is a narrow, read-only projection for external consumers (SCP, Cloud Foundation, SOLAR), each scoped server-side to a curated field set. Both run from the same image, selected by command — not by build (see §09).

Authentication

The scheme is unchanged from legacy — HTTP Basic over TLS. The internal API checks a single credential pair; the partner API accepts one credential pair per partner and adds role-based authorization (require_roles): each partner sees only its own resources, with the M4W role implicitly allowed everywhere. The four partner accounts and the role model are identical, so existing credentials keep working — the rewrite only refactors the implementation (credentials come from typed settings; roles are a StrEnum).

⚠ One auth surface did shrink

Legacy had a second internal credential pair for a dedicated backfill endpoint. That route and its credential are gone in the rewrite (see §12), so the internal API is down to a single credential pair.

Two services, one database. Legacy (compat) keeps serving existing consumers; the rewrite serves migrated ones.

Part II
Compatibility changes
The bridge: one schema migration, a surgical legacy code patch, and a zero-downtime rollout (deploy order + Helm).

05Database & the no-op deploy

The heart of the design: deploy the rewrite without running a single migration against the live schema.

Alembic only ever compares a revision id string. We exploit that. Production sits at the real legacy head 5e67ef3550c0. We author one new legacy migration — 73aee50345b2, the v3 compatibility migration — and apply it; production is now stamped 73aee50345b2. The rewrite collapses its entire schema into a single squashed baseline migration that reuses that exact id. So when the rewrite runs alembic upgrade head, Alembic sees the DB is already at head and does nothing.

The rewrite's baseline migration

The baseline is a single migration holding the full schema as explicit, frozen DDL (op.create_table(...), autogenerated then frozen — not create_all, so later migrations apply cleanly), plus the updated_at trigger and its function (triggers aren't part of the ORM models, so they're recreated explicitly). On production it never executes; on a fresh database — local, a new region, a throwaway test DB — it builds the complete schema in one step.

06Schema changes (the shared DB)

The schema half of the compat commit — every change lives in the one 73aee50345b2 migration, so the live schema matches what the rewrite expects. This is the complete list.

They fall into three groups: identity & referential integrity — clusterlock is re-keyed from the cluster name to the integer cluster_id, and child rows cascade-delete with their parent cluster; invariants the apps rely on — columns that were always populated in practice become NOT NULL, and a feature's name becomes unique; and cleanup — the superseded kubedownscaler_* columns are dropped and ad-hoc unique constraints are renamed to a consistent …_uc form.

Area Before After Why
clusterlock PK on cluster_name PK on cluster_id, cluster_name dropped, FK ON DELETE CASCADE Integer FK identity; locks die with their cluster (orphans deleted first)
cluster kubedownscaler_downscale_period, kubedownscaler_upscale_period dropped Superseded by uptime_period
feature.type nullable NOT NULL (backfilled 'optional') Every feature has a type (namespaced was already NOT NULL)
feature uniqueness none unique (name) → feature_name_uc Feature name is the identity
cluster.account_name NOT NULL nullable Not always known
NOT NULL added cluster.status, clusterfeature.enabled, release.reserved_namespaces, and cluster_id on additionalnodepool / clientnamespace / clientotlpendpoint Invariants the apps rely on (each backfilled first)
unique constraints cluster_repository_key, teamswebhook_webhook_id_key renamed to …_uc Consistent naming
child FKs plain ON DELETE CASCADE node pools, namespaces, OTLP endpoints, cluster/release features all delete with their parent
▶ Note · safe ordering & the operations exception

Every backfill runs before its NOT NULL, and the orphan-lock cleanup runs before the re-key, so the migration is safe on live data. One deliberate exception to the cascade rule: on a fresh DB the rewrite keeps operation.cluster_id as ON DELETE SET NULL — operations survive as an audit trail when a cluster is deleted, rather than vanishing with it.

07The legacy compatibility patch

The code half of the same commit. Legacy (artifact ②) must keep working against the reshaped schema. The patch is small and surgical — behavior is preserved; the code only adapts to the new column shapes and contract. When you read the legacy diff, this is every change and why.

Theme What changed in legacy Why
clusterlock re-key Model, repository, and lock queries key on cluster_id; lock_cluster/unlock_cluster take a Cluster object; reads expose cluster_name via a new ClusterLockRead model that joins back to cluster. The PK changed from name to id, but the API still returns the human-readable name.
kubedownscaler drop The cluster repository strips the two removed kubedownscaler_* fields before persisting. Columns no longer exist; old clients may still send them.
nullable normalization account_name and the AWS/Azure string fields default to None in the models; the repository coerces empty strings to NULL on write. Those columns are now nullable — both apps must agree that "unset" means NULL, not "".
feature null-coercion before-validators on FeatureBase coerce type: null → optional and namespaced: null → false. Those columns are now NOT NULL; an explicit JSON null would otherwise insert SQL NULL and raise IntegrityError.
feature dedup → constraint The feature repository drops its app-level "find existing by name" check (and lists features ascending by id). The new feature_name_uc unique constraint enforces uniqueness at the DB level.
partner projections SCP/SOLAR response models make azure_vnet_name, azure_vnet_resource_group and account_name optional. Those underlying columns can now be NULL; required fields would fail serialization.
OTLP auth alignment The OTLP auth model drops the header_key field and its "required when type=header" validator. The rewrite's OTLP auth shape doesn't carry it; both apps must serialize the auth JSON identically.
release lifecycle The PATCH /v1/releases/{id} endpoint, its ReleaseUpdateRequest, and the repo's update_release are removed. The rewrite has no release-update endpoint — releases are create / read / delete only; legacy is aligned so both present the same release lifecycle.
✓ The point of the patch

It is a thin compatibility shim, not a refactor: legacy accepts the same requests, normalizes deprecated/null fields, and writes only what the shared schema allows. The rewrite owns all new behavior; legacy just keeps the lights on for un-migrated consumers.

08Deploy sequencing

Order matters. The no-op deploy is only safe once the shared DB is already at 73aee50345b2.

  1. Ship the compat-patched legacy code (artifact ②) to all replicas
    clusterlock keyed by cluster_id; kubedownscaler fields stripped on write; feature.type/namespaced null-coercion; nullable-field normalization. The code must tolerate the new schema before the schema changes.
  2. Apply the 73aee50345b2 migration to the shared DB
    Only after the patch rollout completes, and after confirming no pre-patch legacy instance is still live. First verify there are no orphan clusterlock rows (locks whose cluster_name has no matching cluster) — the migration deletes them.
  3. Deploy the rewrite (artifact ③)
    Its Helm pre-sync job runs alembic upgrade head — a no-op, because the baseline id already equals the prod head. The image carries alembic + migrations/ (the migrations dependency group) only so fresh environments can bootstrap; the API itself never imports them.
  4. Rewrite owns the schema; legacy is frozen
    New migrations live in the rewrite, chained from 73aee50345b2, kept backward-compatible while legacy still runs.
  5. Consumers migrate in waves → retire legacy
    When usage drops to zero, the legacy service and SDK are removed.

09Deployment & Helm

One chart, one image, three runtime roles — pointed at the database that already exists.

The chart was renamed clusters-4wm → clusters-metadata-api (and bumped 1.5.0 → 3.0.0), but the template set is unchanged — deployment, job, service, ingress, hpa, servicemonitor, certificates and external-secrets. So the deploy topology is familiar; what changed is how one chart now produces three things.

Three roles from the same image, chosen by command

The deployment template applies .Values.api.command verbatim to the container, and the init-db Job applies .Values.initDb.command — so the role is pure configuration, no rebuild:

Role Command (values) Selected by
Internal API python -m uvicorn app.main:app base values.yaml
Partner API python -m uvicorn app.partner:app values-partner*.yaml overlay (own name, initDb + ServiceMonitor off)
Migration job python -m alembic upgrade head initDb.enabled — an ArgoCD PreSync hook

The migration job is an ArgoCD PreSync hook: it runs alembic upgrade head before the app syncs. On production that's the no-op from §05; on a fresh database it builds the schema first. The partner overlay disables it (initDb.enabled: false) and turns off the ServiceMonitor (the read-only partner app exposes no /metrics). Legacy did the same thing, just through Poetry (poetry run uvicorn api.m4w.main:app).

✗ Point at the existing RDS, not a new one

This is the single most important values change. During coexistence the rewrite must talk to the same physical database the legacy app uses. The values-int/prod files set database.external: true, database.name: clusters4wm, and database.host to the legacy RDS (clusters-4wm-{int,prod}-master…rds.amazonaws.com) — not a new clusters-metadata-api-* instance. New service, old database.

Credentials and DB secrets resolve through External Secrets from a per-service Vault path (applications/clusters-metadata-api-{env}/…), so the two services manage their own credentials while sharing one RDS. Everything else the app needs is injected as environment by the chart — DB_HOST, DB_NAME, DB_USERNAME/PASSWORD, APP_ENVIRONMENT, APP_VERSION, ADGR_GROUP_REGEX, and the basic-auth pairs — and read on the app side through a typed Settings object (no config files baked into the image).

Two smaller niceties: probes moved to the Kubernetes-standard /healthz (liveness + startup) and /readyz (readiness), where legacy used a single /health; and the backfill credential secret is gone along with its route.

10Operating rules during coexistence

✗ Do not ship a legacy schema migration after cutover

Legacy is frozen once the rewrite owns the schema. A new legacy migration would advance the shared alembic_version to a revision the rewrite's squashed history doesn't contain, and the rewrite's alembic upgrade head would fail with Can't locate revision …, blocking its deploys.

⚠ Keep coexistence migrations backward-compatible

While both services run, the rewrite's new migrations must be additive (new nullable columns, new tables) so the frozen legacy app keeps working. Breaking changes wait until legacy is retired.

⚠ Helm values must point at the legacy RDS

During coexistence, database.host in values-int/prod must stay on the clusters-4wm-*-master RDS with DB name clusters4wm. Pointing the rewrite at a fresh clusters-metadata-api-* instance would split the data across two databases and silently break legacy consumers.

⚠ Check for orphan locks before applying 73aee50345b2

The clusterlock re-key deletes rows whose cluster_name has no matching cluster. Verify there are none (or none actively locked) before applying the migration to live prod.

▶ Authoring future migrations

Run uv run alembic upgrade head on a dev DB first, change the models, then alembic revision --autogenerate — it emits only the delta against the at-head schema. Always review the result: autogenerate does not detect renames, triggers/functions, or (by default) server-default changes.

11Code structure: before & after

The rewrite is organized into explicit layers with clear boundaries between transport, logic, and persistence.

Legacy — src/api

api/m4w v1 + v2 routers
api/partner scp / solar
api/shared models · repos · enums
SQLModel table = schema = validator

Rewrite — src/app

api m4w · partner · /v3
schemas Pydantic v2 I/O
services business logic + txn boundary
repositories SQLAlchemy 2 data access
models typed ORM (Mapped)
core config · db · security · logging · metrics

Why this shape. Each layer has one job and dependencies point in a single direction — api → services → repositories → models — so HTTP concerns never reach the business logic and SQL never leaks into the routers. Request/response shapes live in schemas (Pydantic v2), deliberately separate from the models (the DB tables): the wire contract and the storage layout evolve independently, and the API can't accidentally expose a column. services own the transaction boundary (commit/rollback happen there, not in routers or repos), and core holds the cross-cutting setup. The cluster aggregate — the only complex one — is itself split into a service / translator / mapper trio so its nested-field syncing stays readable.

Two entrypoints share this codebase: app.main:app (internal) and app.partner:app (partner). They ship as the same image; the running app — or the migration job — is chosen by overriding the container command (Helm api.command, compose command), not by a separate build (see §09).

12API contract & behavior changes

v3 keeps v2's resource shapes — a cluster or feature looks the same on the wire — with a handful of deliberate departures. These are the ones that change observable behavior.

Aspect Legacy New
Surface /v1, /v2 /v3 only
Single-cluster routes by numeric id — /clusters/{id} by name — /v3/clusters/{cluster_name}
List responses mixed: bare arrays + a {count, clusters} wrapper uniform { "count": N, "items": [...] } everywhere
Errors {"detail": "…"} (and {"message": …} on partner) RFC 7807 application/problem+json
Partner empty result 404 {"message": "No clusters found"} 200 + empty list intentional
Partner routes /v1/clusters, /v1/solar/clusters /v3/partners/scp/clusters, /v3/partners/solar/clusters
OpenAPI / SDK spec produced on demand; SDK generated ad-hoc openapi.json committed; deterministic regeneration (sorted keys); SDK generated from it

Behavior a consumer will notice

  • Addressed by name, not id. Single-cluster routes use /v3/clusters/{cluster_name} (and locks/operations hang off the same path); the name is already unique.
  • Uniform list envelope. Every list endpoint returns { "count": N, "items": [...] } — note the key is items, not legacy v2's clusters. Legacy's bare-array endpoints are gone.
  • Aggregation, sub-resource & backfill reads dropped. Gone: /clusters/releases and /clusters/infra-revisions (aggregations), /clusters/{id}/client-otlp-endpoints and /client-namespaces (now part of the cluster representation), /clusters/adgr, the single-cluster /clusters/{id}/status read, and /clusters/{id}/backfill. The status list and bulk update are kept (GET/PATCH /v3/clusters/status) — see the full map in §13.
  • Releases are create / read / delete only. There is no release-update endpoint — a release is replaced, not patched.
  • Empty features is a no-op on write. Sending features: [] on create/patch does not wipe a cluster's features — they are only rebuilt from a non-empty list. This deliberately preserves legacy semantics and protects the shared DB the legacy app still reads.
  • Partner empty result is 200, not 404. An empty match returns { "count": 0, "items": [] } instead of a 404 error body.

Partner API — a real contract change

Internal resource shapes carry over from v2, but the partner projection genuinely changed on the wire. Partner consumers (SCP, CF, SOLAR) must adapt, not just re-point. The selected fields are otherwise unchanged.

AspectLegacyNew
Path /v1/clusters, /v1/solar/clusters /v3/partners/scp/clusters, /v3/partners/solar/clusters
Envelope { count, clusters } (clusters nullable) { count, items } (items always a list)
Item identity no id new id field on every item
Nullable fields required strings SCP azure_vnet_name / azure_vnet_resource_group, SOLAR account_name may be null
Empty result 404 {"message": …} 200 { "count": 0, "items": [] }
Query filters bespoke per-partner filter set shared ClusterListQuery (extra=forbid): SOLAR keeps its filters; SCP loses azure_vnet_name / azure_vnet_resource_group as filters

Error format

Domain errors and transport errors (including auth 401/403) are now RFC 7807 application/problem+json:

# legacy
{ "detail": "Cluster not found" }

# rewrite — Content-Type: application/problem+json
{ "type": "about:blank", "title": "Not Found", "status": 404, "detail": "Cluster not found" }

Request-validation errors (422) are intentionally left in FastAPI's default shape so they keep matching the generated OpenAPI/SDK 422 schema.

Why OpenAPI is checked in. The API contract lives in version control as openapi.json, regenerated deterministically with sorted keys (scripts/generate_openapi_spec.py) so it only changes when the contract really does. Regenerating is a single command, so any change to the public API — and the SDK generated from it (scripts/generate_sdk.sh, sdk-config.yml) — shows up as a reviewable diff in the same PR. In legacy the spec existed only at runtime, so contract drift stayed invisible until a consumer hit it.

Same entrypoint model, better build. Both legacy and the rewrite ship a single image that can run the internal app, the partner app, or the migration job — the role is chosen by overriding the container command (compose / Helm), never by separate images or build targets. That didn't change. Legacy's Dockerfile simply hard-codes the internal app as its default CMD and relied on command overrides for the rest — the rewrite does the same, just documents it. What actually changed is image hygiene: legacy is a single-stage image that pip installs Poetry and copies the source tree in; the rewrite's multi-stage uv build keeps Poetry/build tooling and dev dependencies out of the runtime layer, installs the package into a venv instead of copying source, and adds a healthcheck.

ORM modeling, before & after. A legacy SQLModel class was the table, the validator, and the API schema all at once. The rewrite splits them: a typed SQLAlchemy model is just the table, and request/response shapes live in separate Pydantic v2 schemas (so the wire contract and the storage layout evolve independently).
