▶ TL;DR

clusters-4wm is being replaced by a clean rewrite, clusters-metadata-api. Both run in parallel against the same PostgreSQL database while consumers migrate to the new /v3 API and SDK. To make the existing schema fit the new app, we applied a one-time compatibility migration (73aee50345b2) to legacy and patched a little legacy code. The new app ships a single squashed baseline migration whose id equals the production Alembic head, so deploying it runs no migration on the live DB. After cutover the rewrite owns the schema and legacy is frozen.

01Why the rewrite

The legacy clusters-4wm service accumulated patterns that were costly to evolve. Because the domain is essentially CRUD over managed-cluster metadata, a clean rewrite was faster and safer than incrementally untangling the old codebase. The rewrite also takes a new canonical identity:

AreaLegacyNew
Service / chartclusters-4wmclusters-metadata-api
API surface/v1, /v2/v3
Generated SDKclusters-4wm-clientclusters-metadata-sdk

The hard constraint: we cannot do a big-bang switch. Consumers migrate in waves over months, so the legacy and new services must run simultaneously against one shared database and stay compatible the whole time.

02The service & its API surface

What the application actually is, before we get to how it migrates.

clusters-metadata-api is a CRUD service over managed-cluster metadata — clusters and the things attached to them (features, locks, operations, releases). One codebase ships two FastAPI apps:

Internal API — app.main:app

/v3/clusters CRUD · status
/v3/features list · create · delete
/v3/locks acquire / release per cluster
/v3/operations per-cluster operation log
/v3/releases release catalog

Partner API — app.partner:app

read-only cluster views
SCP · SOLAR per-partner projections
role-filtered {count, items}

The internal API is the full read/write management surface used by 4Wheels tooling. The partner API is a narrow, read-only projection for external consumers (SCP, Cloud Foundation, SOLAR). Both run from the same image, selected by command (see §04).

Authentication

Auth is unchanged from legacy — HTTP Basic over TLS. The internal API checks a single credential pair; the partner API accepts one credential pair per partner and adds role-based authorization (require_roles): each partner sees only its own resources, with the M4W role implicitly allowed everywhere. The rewrite only refactors the implementation (credentials come from typed settings, roles are a StrEnum) — the scheme, the four partner accounts, and the role model are identical, so existing credentials keep working.

03Coexistence architecture

Two services, one database. Legacy keeps serving existing consumers; the rewrite serves migrated ones.

04Code structure: before & after

The rewrite is organized into explicit layers with clear boundaries between transport, logic, and persistence.

Legacy — src/api

api/m4w v1 + v2 routers
api/partner scp / solar
api/shared models · repos · enums
SQLModel mixed model/schema

Rewrite — src/app

api m4w · partner · /v3
schemas Pydantic v2 I/O
services business logic + UoW
repositories SQLAlchemy 2 data access
models typed ORM (Mapped)
core config · db · security · logging · metrics

Why this shape. Each layer has one job and dependencies point in a single direction — api → services → repositories → models — so HTTP concerns never reach the business logic and SQL never leaks into the routers. Request and response shapes live in schemas (Pydantic v2), kept deliberately separate from the models (the DB tables): the wire contract and the storage layout can evolve independently, and the API can't accidentally expose a column. services own the unit-of-work transaction boundary, and core holds the cross-cutting setup (config, DB engine, security, logging, metrics). The legacy app mixed these — SQLModel classes were both the table and the API schema, and data access sat alongside routing — which is what made it costly to change.

Two entrypoints share this codebase: app.main:app (the internal M4W API) and app.partner:app (the external, read-only partner API). They ship as the same image — the running app is chosen by command, not by build (see §10).

05Database & migration strategy

The heart of the design: deploy the rewrite without touching the already-migrated production schema.

Production databases are already stamped at the legacy Alembic head 73aee50345b2. The rewrite collapses the entire legacy chain into a single squashed baseline migration that reuses that exact id. So when the rewrite runs alembic upgrade head, Alembic sees the DB is already at head and does nothing.

The baseline migration

The baseline is a single migration holding the full schema as explicit, frozen DDL (op.create_table(...)), plus the updated_at trigger the schema depends on (triggers aren't part of the ORM models, so they're created explicitly). On production it never executes; on a fresh database — CI, local, or a new region — it builds the complete schema in one step.

06Deploy sequencing

Order matters. The no-op deploy is only safe if the shared DB is already at 73aee50345b2.

  1. Ship the patched legacy code to all replicas
    clusterlock keyed by cluster_id, kubedownscaler fields dropped, feature.type/namespaced null-coercion.
  2. Apply the legacy 73aee50345b2 migration to the shared DB
    Only after the patch rollout completes. First verify there are no orphan clusterlock rows (rows whose cluster_name has no matching cluster) — the migration deletes them.
  3. Deploy the rewrite
    Baseline id = prod head ⇒ alembic upgrade head is a no-op. The image carries alembic + migrations/ only so fresh environments can still bootstrap.
  4. Rewrite owns the schema; legacy is frozen
    New migrations live in the rewrite, chained from 73aee50345b2, kept backward-compatible while legacy still runs.
  5. Consumers migrate in waves → retire legacy
    When usage drops to zero, the legacy service and SDK are removed.

07Schema changes (the shared DB)

All applied to production in one place — the legacy 73aee50345b2 migration — so the live schema matches what the new app expects.

They fall into three groups: identity & referential integrity — clusterlock is re-keyed from the cluster name to the integer cluster_id, and children now cascade-delete with their parent cluster; invariants the app relies on — columns that were always populated in practice become NOT NULL, and a feature's name becomes unique; and cleanup — the superseded kubedownscaler_* columns are dropped and ad-hoc unique constraints renamed to a consistent …_uc form. 08What we adapted in the legacy app

The legacy service has to keep working against the post-73aee50345b2 schema. The patch is deliberately small and surgical — its behavior is unchanged; the code only adapts to the new column shapes. When you read the legacy diff, this is what each change is and why.

ChangeWhat & why
The 73aee50345b2 migrationThe schema half: every change in §7, carried as one compatibility migration. Everything below is the code half — legacy catching up to that schema.
clusterlock by cluster_idLegacy model, repository, and lock queries now key on cluster_id; the lock read model derives cluster_name via a join.
Drop kubedownscaler on writeThe legacy cluster repository strips the removed kubedownscaler_* fields before persisting.
feature.type / namespaced null-coercionAfter feature.type became NOT NULL, an explicit type: null (or omitted namespaced) would insert SQL NULL and fail. before-validators coerce null→default and default namespaced to False.

09API contract & behavior changes

v3 keeps v2's resource shapes — a cluster or feature looks the same on the wire — with a handful of deliberate departures. These are the ones that change observable behavior.

AspectLegacyNew
Surface/v1, /v2/v3 only
Errors{"detail": "<string>"}RFC 7807 application/problem+json
Partner empty result404 {"message": "No clusters found"}200 + empty list intentional change
OpenAPI / SDKspec produced on demand; SDK generated ad-hocopenapi.json committed to the repo; CI fails if it drifts from the code

Behavior a consumer will notice

  • Addressed by name, not id. Single-cluster routes are /v3/clusters/{cluster_name} (legacy used the numeric id); the name is already unique.
  • Uniform list envelope. Every list endpoint returns { "count": N, "items": [...] }. Legacy v2 mixed bare arrays (e.g. GET /v2/features) with wrapped ones.
  • Aggregation & backfill endpoints dropped. The legacy /clusters/releases, /clusters/infra-revisions and backfill routes are gone — the same data is read off the cluster resource itself.
  • Empty features is a no-op on write. Sending features: [] on create/patch does not wipe a cluster's features — they are only rebuilt from a non-empty list. This deliberately preserves legacy semantics and protects the shared DB the legacy app still reads.
  • Partner empty result is 200, not 404. An empty match returns { "count": 0, "items": [] } instead of a 404 error body.

Error format

Domain errors and transport errors (including auth 401/403) are now RFC 7807 application/problem+json:

# legacy
{ "detail": "Cluster not found" }

# rewrite — Content-Type: application/problem+json
{ "type": "about:blank", "title": "Not Found", "status": 404, "detail": "Cluster not found" }

Request-validation errors (422) are intentionally left in FastAPI's default shape so they keep matching the generated OpenAPI/SDK.

Why OpenAPI is checked in. The API contract lives in version control as openapi.json, regenerated with sorted keys (scripts/generate_openapi_spec.py) so it only changes when the contract really does. CI regenerates it and fails the build if it differs from the committed copy — so no change can alter the public API, and the SDK generated from it, without that showing up as a reviewable diff in the same PR. In legacy the spec existed only at runtime, so contract drift stayed invisible until a consumer hit it.

ORM modeling, before & after. A legacy SQLModel class was the table, the validator, and the API schema all at once. The rewrite splits them: a typed SQLAlchemy model is just the table, and request/response shapes live in separate Pydantic v2 schemas (so the wire contract and the storage layout evolve independently).

Legacy — SQLModel (table + schema)

class FeatureBase(SQLModel):
    name: Annotated[str, AfterValidator(...)]
    type: Optional[str] = FeatureType.OPTIONAL
    dependencies: Optional[List[LowercasedStr]] = Field(
        default=[], sa_column=Column(JSON, nullable=False))
    namespaced: Optional[bool] = Field(default=False, ...)

class Feature(FeatureBase, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    clusters: List["ClusterFeature"] = Relationship(...)

Rewrite — SQLAlchemy 2 (table only)

class Feature(Base):
    __tablename__ = "feature"
    __table_args__ = (UniqueConstraint("name", name="feature_name_uc"),)

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]
    type: Mapped[str] = mapped_column(default="optional")
    dependencies: Mapped[list[str]] = mapped_column(JSON, ...)
    namespaced: Mapped[bool] = mapped_column(default=False, ...)

    cluster_features: Mapped[list[ClusterFeature]] = relationship(...)

11Operating rules during coexistence

✗ Do not ship a legacy schema migration after cutover

Legacy is frozen once the rewrite owns the schema. A new legacy migration would advance the shared alembic_version to a revision the rewrite's squashed history doesn't contain, and the rewrite's alembic upgrade head would fail with Can't locate revision …, blocking its deploys.

⚠ Keep coexistence migrations backward-compatible

While both services run, the rewrite's new migrations must be additive (new nullable columns, new tables) so the frozen legacy app keeps working. Breaking changes wait until legacy is retired.

⚠ Check for orphan locks before applying 73aee50345b2

The clusterlock re-key deletes rows whose cluster_name has no matching cluster. Verify there are none (or none that are actively locked) before applying it to live prod.

▶ Future migrations

Run uv run alembic upgrade head on a dev DB first, change the models, then alembic revision --autogenerate — it emits only the delta against the at-head schema. Always review the result: autogenerate does not detect renames, triggers/functions, or (by default) server-default changes.
